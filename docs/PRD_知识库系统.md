## 1. 问题陈述

### 1.1 核心痛点

**问题**: 知识工作者面临信息分散在多个系统的困境,导致检索效率低下和知识孤岛。

**具体表现**:
1. **本地笔记与内网 Wiki 割裂**: 个人笔记存在本地 Markdown 文件,公司知识库在内网 Wiki 系统,需要在多个界面之间切换查找信息
2. **内网 Wiki 无法离线访问**: 离开公司网络后无法查询公司知识库,影响远程办公效率
3. **检索效率低**: 传统文本搜索无法理解语义,找到相关但关键词不匹配的内容困难
4. **AI 工具无法访问私有知识**: 使用 Claude CLI 时,无法直接查询本地笔记和内网 Wiki 内容,需要手动复制粘贴

**为什么这个问题重要?**
- **第一层 Why**: 信息检索效率直接影响知识工作者的生产力
- **第二层 Why**: 知识孤岛导致重复劳动(同一问题需要多次查找/询问同事)
- **第三层 Why**: 企业的知识资产无法有效复用,新员工成长周期延长,团队整体效率下降

### 1.2 目标用户

**主要用户**: 需要频繁查阅技术文档、业务知识的知识工作者

**典型画像**:
- 软件工程师:需要查询公司内部技术规范、API 文档、最佳实践
- 产品经理:需要检索历史需求文档、业务流程、竞品分析
- 数据分析师:需要查找数据字典、分析方法论、报告模板

**使用频率**: 每天 10-50 次检索操作

### 1.3 不解决的成本

- **时间成本**: 每次检索平均耗时 5-15 分钟(打开多个系统 + 多次搜索),每天浪费 1-2 小时
- **机会成本**: 无法在 AI 对话中直接引用私有知识,降低 AI 辅助工作的价值
- **知识流失**: 离职员工的隐性知识未能有效沉淀,团队知识资产持续贬值

## 2. 目标与成功指标

### 2.1 业务目标

**核心目标**: 构建统一的个人知识库系统,实现本地笔记 + 内网 Wiki 的无缝检索,并通过 MCP 协议集成到 Claude CLI 工作流。

| 目标 | 可衡量指标 | 目标值 | 测量窗口 |
|------|-----------|--------|---------|
| **检索效率** | 平均检索响应时间 | < 500ms | 上线后 30 天 |
| **内容覆盖** | Wiki 爬取成功率 | > 95% | 每次爬取任务 |
| **系统可用性** | MCP 服务调用成功率 | > 99% | 上线后 30 天 |
| **检索质量** | 混合检索 Top-5 命中率 | > 80% | 人工抽样 100 次查询 |
| **用户采纳** | 日活跃查询次数 | > 20 次/天 | 上线后 30 天 |

### 2.2 技术目标

- 支持 10,000+ 文档的向量索引(单机)
- 并发爬取时控制请求速率(避免触发内网 WAF)
- MCP 服务响应时间 < 200ms(P95)

## 3. 非目标(本次迭代不做)

**明确边界,保护团队注意力:**

❌ **不支持移动端**: 仅提供本地命令行工具,不开发 iOS/Android 客户端
❌ **不支持多用户协作**: 单用户本地部署,不涉及权限管理、实时协作编辑
❌ **不支持 MCP 写入操作**: MCP 服务仅提供查询接口,不支持创建/编辑笔记
❌ **不支持高级权限管理**: 不区分文档的可见性层级,所有内容平等索引
❌ **不支持云端同步**: 所有数据存储在本地,不提供云端备份/同步功能
❌ **不支持实时增量爬取**: 内网 Wiki 爬取采用定时全量/手动触发,不实现 Webhook 增量更新
❌ **不支持多语言 NLP**: 仅支持中文/英文内容,不处理小语种文档

**为什么不做?**
- **聚焦核心价值**: 先验证"统一检索"的核心价值,避免过度工程化
- **控制复杂度**: 多用户协作、云端同步会显著增加系统复杂度和开发周期
- **技术风险**: 实时增量爬取需要深度定制 Wiki 系统,通用性差

## 4. 用户故事与验收标准

### 故事 1: 创建和管理本地笔记

**作为** 知识工作者
**我想要** 创建 Markdown 格式的笔记并支持全文搜索
**以便** 快速记录和查找个人知识

**验收标准**:
- **Given** 我打开笔记管理界面
  **When** 我创建一篇新笔记并输入 Markdown 内容
  **Then** 笔记自动保存到本地,支持实时预览
- **Given** 我有 100+ 篇笔记
  **When** 我输入关键词进行搜索
  **Then** 在 500ms 内返回相关笔记列表(按相关度排序)

---

### 故事 2: 配置内网 Wiki 爬取规则

**作为** 知识工作者
**我想要** 通过可视化界面配置 Wiki 爬取规则(登录方式 + XPath 路径)
**以便** 自动同步内网知识到本地知识库

**验收标准**:
- **Given** 我访问爬虫配置页面
  **When** 我选择登录方式(扫码/API Token)并完成认证
  **Then** 系统验证登录状态并提示"认证成功"
- **Given** 我已完成登录
  **When** 我输入目标页面 URL 和内容提取 XPath
  **Then** 系统预览提取结果(前 500 字)供我确认
- **Given** 我启用递归爬取
  **When** 我设置递归深度为 3 层
  **Then** 系统遍历所有 `<a>` 链接并显示预计爬取页面数

---

### 故事 3: 执行 Wiki 爬取任务

**作为** 知识工作者
**我想要** 手动触发 Wiki 爬取任务并监控进度
**以便** 确保内网知识已同步到本地

**验收标准**:
- **Given** 我已配置爬取规则
  **When** 我点击"开始爬取"
  **Then** 系统显示实时进度(已爬取/总页面数、当前页面 URL、失败数)
- **Given** 爬取任务运行中
  **When** 遇到 HTTP 429(限流)错误
  **Then** 系统自动等待 60 秒后重试,并在日志中记录
- **Given** 爬取任务完成
  **When** 我查看爬取报告
  **Then** 显示成功率、失败页面列表、总耗时

---

### 故事 4: 混合检索知识库

**作为** 知识工作者
**我想要** 使用自然语言查询知识库(混合检索)
**以便** 找到语义相关但关键词不匹配的内容

**验收标准**:
- **Given** 我在搜索框输入"如何优化数据库查询性能"
  **When** 系统执行混合检索
  **Then** 返回 Top-10 结果,包含:
    - 文档标题
    - 相关度得分(全文 + 语义加权)
    - 内容摘要(高亮匹配部分)
    - 来源标签(本地笔记 / Wiki)
- **Given** 检索结果包含 Wiki 页面
  **When** 我点击结果
  **Then** 在新标签页打开原始 Wiki 页面 URL

---

### 故事 5: 通过 Claude CLI 查询知识库

**作为** Claude CLI 用户
**我想要** 在对话中直接查询本地知识库
**以便** 让 AI 引用我的私有知识回答问题

**验收标准**:
- **Given** MCP 服务已启动
  **When** 我在 Claude CLI 中询问"公司的代码审查流程是什么"
  **Then** Claude 自动调用 MCP 查询接口,返回相关文档片段并引用来源
- **Given** 查询无相关结果
  **When** MCP 返回空结果
  **Then** Claude 提示"未在本地知识库中找到相关内容"

### 5.2 关键交互流程

#### 流程 1: Wiki 爬取配置与执行

```
用户 → 配置登录方式(扫码/Token) → 系统验证登录 →
用户输入目标 URL + XPath → 系统预览提取结果 →
用户确认并启动爬取 → 系统并发爬取(限流控制) →
内容预处理(清洗/分块) → 写入向量数据库 →
生成爬取报告
```

#### 流程 2: 混合检索执行

```
用户输入查询 →
  ├─ 全文检索(BM25) → Top-K1 结果
  └─ 语义检索(向量相似度) → Top-K2 结果
      ↓
  融合排序(RRF/加权) → Top-N 最终结果 → 返回用户
```

#### 流程 3: MCP 查询集成

```
Claude CLI 用户提问 →
Claude 判断需要查询知识库 →
调用 MCP 查询接口(传递查询文本) →
MCP 服务执行混合检索 →
返回 Top-5 文档片段 + 元数据 →
Claude 引用知识库内容生成回答
```

### 5.3 关键设计决策

#### 决策 1: 为什么选择混合检索?

**问题**: 单一检索方式无法满足不同查询场景
**分析**:
- 全文检索(BM25)擅长精确关键词匹配,但无法理解语义
- 语义检索(向量)擅长理解意图,但对专有名词/代码敏感度低

**决策**: 采用混合检索(BM25 + 向量语义),通过 RRF(Reciprocal Rank Fusion)融合排序
**权衡**: 增加系统复杂度(需要维护两套索引),但显著提升检索召回率(预计 +30%)

#### 决策 2: 为什么 MCP 服务只提供查询接口?

**问题**: 是否需要支持通过 Claude 创建/编辑笔记?
**分析**:
- **Why 1**: 核心价值是"检索",而非"创作" - 用户已有笔记编辑工具
- **Why 2**: 写入操作涉及数据一致性、权限校验,显著增加复杂度
- **Why 3**: Claude CLI 的主要场景是"问答",而非"内容管理"

**决策**: 本次迭代仅提供只读查询接口
**未来可能**: 如果用户反馈强烈需要"AI 辅助写作",可在 v2 版本考虑添加

#### 决策 3: 为什么选择 Chroma 作为向量数据库?

**问题**: Chroma vs Qdrant vs LanceDB?
**分析**:

| 维度 | Chroma | Qdrant | LanceDB |
|------|--------|--------|---------|
| **部署复杂度** | 低(嵌入式) | 中(需独立进程) | 低(嵌入式) |
| **性能(10K文档)** | 中 | 高 | 高 |
| **社区成熟度** | 高 | 高 | 中 |
| **Node.js 支持** | 好 | 好 | 一般 |

**决策**: 选择 Chroma
**原因**:
1. 嵌入式部署,用户无需额外配置独立数据库
2. 社区活跃,文档完善,降低团队学习成本
3. 10K 文档规模下性能足够(后续可迁移至 Qdrant)

#### 决策 4: 爬虫并发控制策略

**问题**: 如何平衡爬取速度与避免触发内网 WAF?
**分析**:
- 并发过高 → 触发限流 → 爬取失败率上升
- 并发过低 → 爬取耗时过长 → 用户体验差

**决策**: 采用**令牌桶算法** + **自适应退避**
- 初始并发度: 5 个请求/秒
- 遇到 429 错误: 指数退避(60s → 120s → 240s)
- 成功率 > 95%: 逐步提升并发度(最大 10/秒)

## 8. 附录

### 8.1 术语表

| 术语 | 定义 |
|------|------|
| **混合检索** | 结合全文检索(BM25)和语义检索(向量相似度)的检索方法 |
| **RRF(Reciprocal Rank Fusion)** | 一种融合多个排序结果的算法,通过倒数排名加权 |
| **MCP(Model Context Protocol)** | Anthropic 定义的协议,用于外部工具与 Claude 集成 |
| **XPath** | XML 路径语言,用于在 HTML/XML 文档中定位元素 |
| **令牌桶算法** | 限流算法,通过固定速率发放令牌控制请求速率 |
| **Chunk** | 文档分块单元,用于向量化和检索的最小粒度 |
