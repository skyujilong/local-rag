### 1️⃣ PRD_知识库系统.md (Product Manager)

**文档结构** (7大部分):

1. **问题陈述**
   - 核心痛点:信息分散、检索效率低、AI工具无法访问私有知识
   - 目标用户:软件工程师、产品经理、数据分析师
   - 不解决的成本:每天浪费1-2小时检索时间

2. **目标与成功指标**
   - 业务目标:构建统一知识库,实现本地笔记+内网Wiki无缝检索
   - 成功指标:
     - 检索时间从5-15分钟降至<30秒
     - Claude CLI查询成功率>95%
     - 用户NPS>50

3. **功能需求** (5个用户故事)
   - US-001:作为知识工作者,我希望在统一界面搜索笔记和Wiki
   - US-002:作为用户,我希望通过扫码或Token登录内网Wiki
   - US-003:作为用户,我希望使用语义搜索找到相关内容
   - US-004:作为开发者,我希望在Claude CLI中直接查询知识库
   - US-005:作为管理员,我希望配置爬虫规则和监控任务状态

4. **RICE优先级评分**
   - 本地笔记管理:RICE=72(高优先级)
   - Wiki爬取引擎:RICE=64(高优先级)
   - 向量知识库:RICE=60(中优先级)
   - MCP服务:RICE=48(中优先级)

5. **技术方案概述**
   - 前端:Vue3 + Element Plus
   - 后端:Node.js + Express
   - 向量数据库:LanceDB/Chroma
   - 爬虫:Puppeteer/Cheerio + XPath

6. **风险与依赖** (6个风险项)
   - 风险1:内网Wiki登录机制复杂(扫码/双因素认证)
   - 风险2:XPath规则变化导致爬取失败
   - 风险3:向量数据库性能不足(大数据量场景)
   - 风险4:MCP协议更新导致兼容性问题
   - 风险5:爬虫并发过高对内网系统造成压力
   - 风险6:嵌入模型成本过高(OpenAI API)

7. **产品路线图**
   - MVP阶段(2周):本地笔记+全文搜索
   - Phase 1(4周):Wiki爬取+语义搜索
   - Phase 2(2周):MCP服务集成
   - Phase 3(持续):性能优化+监控

### 3️⃣ MCP服务设计_知识库系统.md (MCP Builder)

**文档结构** (36KB,包含完整TypeScript实现):

1. **概述**
   - MCP架构图:Claude CLI ↔ MCP Server(stdio) ↔ 后端API(HTTP)
   - 核心特性:多源检索、混合搜索、类型安全、容错设计

2. **4个MCP工具定义**

   **工具1:search_notes** (搜索本地笔记)
   - 输入参数:
     - `query: string` (必填) - 搜索关键词
     - `limit?: number` (可选,默认10) - 返回结果数量(1-100)
   - 输出:JSON数组,包含笔记标题、内容片段、标签、创建时间
   - Zod Schema:
     ```typescript
     { query: z.string(), limit: z.number().min(1).max(100).optional() }
     ```

   **工具2:query_wiki** (查询Wiki内容)
   - 输入参数:
     - `query: string` (必填)
     - `source_filter?: string[]` (可选) - 筛选特定Wiki来源
     - `limit?: number` (可选,默认10)
   - 输出:Wiki页面标题、摘要、URL、最后更新时间

   **工具3:semantic_search** (语义搜索)
   - 输入参数:
     - `query: string` (必填) - 自然语言查询
     - `top_k?: number` (可选,默认5) - 返回Top-K结果
     - `score_threshold?: number` (可选,默认0.7) - 相似度阈值(0-1)
   - 输出:文档ID、相似度分数、来源、内容片段

   **工具4:hybrid_search** (混合检索)
   - 融合全文检索和语义检索
   - 输入参数:
     - `query: string` (必填)
     - `alpha?: number` (可选,默认0.5) - 全文检索权重(0-1)
     - `top_k?: number` (可选,默认10)
   - 输出:融合后的排序结果(RRF算法)

## 🎯 技术方案汇总

### 核心技术栈

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| **前端** | Vue 3.4 + Vite + TypeScript | Composition API灵活、Vite构建快速、TypeScript类型安全 |
| **UI组件** | Element Plus 2.5 | 组件丰富、文档完善、支持Tree-shaking |
| **Markdown编辑器** | Vditor 3.10 | 轻量级(~200KB)、支持3种模式、自带自动保存 |
| **状态管理** | Pinia 2 | Vue3官方推荐、API简洁、TypeScript支持好 |
| **后端** | Node.js + Express | JavaScript全栈、生态成熟、开发效率高 |
| **爬虫** | Cheerio + xpath | 轻量级HTML解析、XPath查询灵活 |
| **并发控制** | p-queue | 轻量级、无外部依赖、满足基本需求 |
| **向量数据库** | LanceDB | 嵌入式、轻量级、适合本地部署 |
| **嵌入模型** | OpenAI text-embedding-3-small | 成本低、性能好、易于集成 |
| **全文检索** | MeiliSearch | 轻量级、易部署、搜索速度快 |
| **进程管理** | PM2 | 成熟稳定、日志管理、自动重启 |
| **日志** | Winston | 结构化日志、多种Transport、易于扩展 |

### 关键技术决策(ADR)

**ADR-001: 选择LanceDB作为向量数据库**
- **理由**: 嵌入式设计,无需独立服务;支持全文+向量混合检索;资源占用低,适合本地运行
- **权衡**: 生态较新,社区支持有限;未来如需分布式部署可能需迁移到Qdrant
- **状态**: 已接受

**ADR-002: 使用p-queue进行并发控制**
- **理由**: 轻量级,无外部依赖(不需要Redis);API简洁,满足基本限流需求
- **权衡**: 无任务持久化,进程重启后任务丢失;如需任务持久化,考虑迁移到Bull
- **状态**: 已接受

**ADR-003: 优先实现API Token注入,扫码登录作为增强功能**
- **理由**: Token注入实现简单,开发快速;Puppeteer资源占用高,复杂度大
- **权衡**: 用户需要手动获取Token,体验稍差;扫码登录可在MVP后补充
- **状态**: 已接受

**ADR-004: 使用RRF算法融合混合检索结果**
- **理由**: 算法简单,无需调参;对不同检索模式公平;性能稳定
- **权衡**: 不如学习型排序(LTR)精准,但简单场景足够
- **状态**: 已接受

## ⚙️ 核心功能实现要点

### 1. XPath递归爬取引擎
```javascript
// 核心伪代码
async function crawlPage(url, rule, depth, visitedUrls) {
  if (depth > rule.recursiveDepth) return;
  if (visitedUrls.has(url)) return;

  visitedUrls.add(url);
  const html = await fetchPage(url, rule.auth);  // 注入Cookie/Token
  const $ = cheerio.load(html);

  // 提取内容(XPath → CSS Selector转换)
  const content = extractByXPath($, rule.contentXPath);
  await saveContent(url, content);

  // 提取链接并递归
  const links = extractByXPath($, rule.linksXPath);
  for (const link of links) {
    const absoluteUrl = resolveUrl(url, link);
    await taskQueue.add(() => crawlPage(absoluteUrl, rule, depth + 1, visitedUrls));
  }
}
```

### 2. 混合检索RRF算法
```javascript
// RRF(Reciprocal Rank Fusion)
function rrf(fulltextResults, semanticResults, k = 60) {
  const scores = {};

  fulltextResults.forEach((doc, rank) => {
    scores[doc.id] = (scores[doc.id] || 0) + 1 / (k + rank + 1);
  });

  semanticResults.forEach((doc, rank) => {
    scores[doc.id] = (scores[doc.id] || 0) + 1 / (k + rank + 1);
  });

  return Object.entries(scores)
    .sort((a, b) => b[1] - a[1])
    .slice(0, 10)
    .map(([id, score]) => ({ id, score }));
}
```

### 3. 并发控制(p-queue)
```javascript
import PQueue from 'p-queue';

const queue = new PQueue({
  concurrency: 5,        // 最大5个并发
  interval: 500,         // 每500ms处理一个任务(限流)
  intervalCap: 1
});

// 添加任务
queue.add(() => crawlPage(url1, rule, 0));
queue.add(() => crawlPage(url2, rule, 0));

// 监听完成
queue.on('idle', () => {
  console.log('All tasks completed');
});
```

   - 索引优化:SQLite添加索引、LanceDB分片

   - 向量数据库:LanceDB分片存储,加速检索
   - 全文检索:MeiliSearch预构建索引
   - 混合检索:RRF算法,O(n)复杂度,性能稳定

   - 备份策略:定期备份笔记和配置
