## 1. 问题陈述

### 1.1 核心痛点

**背景场景**
- 程序员在日常开发中大量使用 Claude AI 辅助编程
- 公司内部存在大量技术文档、Wiki 知识库、团队协作文档
- 开发者个人也有大量本地笔记、代码片段、学习记录
- Claude AI 无法访问这些私有知识源，导致上下文理解不足

**具体痛点**
1. **知识孤岛问题**
   - 公司技术文档分散在 Confluence、Notion、GitLab Wiki 等平台
   - 个人笔记分散在 Obsidian、VS Code、本地 Markdown 文件
   - Claude AI 无法获取这些专业知识，回答通用化、不精准

2. **上下文成本高昂**
   - 向 Claude 提供背景知识需要手动复制粘贴
   - Token 消耗大，每次对话都要重复说明公司技术栈、架构规范
   - 复杂技术问题需要多轮对话才能建立上下文

3. **知识检索效率低**
   - 遇到问题时需要手动搜索文档、笔记
   - 无法快速定位相关历史解决方案
   - 团队知识传承依赖口口相传

### 1.2 目标用户

**主要用户画像**
- **角色**：软件开发工程师（个人开发者或团队成员）
- **技术背景**：熟悉 Node.js、使用 Claude Code 或 Claude CLI
- **工作场景**：
  - 需要频繁查阅公司内部技术文档
  - 维护个人技术笔记和知识库
  - 使用 Claude AI 辅助代码开发、调试、架构设计
- **痛点**：Claude 缺乏领域知识，回答不够专业化和定制化

**用户规模**
- 初期目标：个人工具，服务单个开发者
- 未来扩展：支持小团队（5-10人）共享知识库

### 1.3 不解决的成本

**如果不使用 devrag-cli**
- **效率损失**：每次开发任务增加 20-30% 的上下文准备时间
- **知识浪费**：历史问题和解决方案无法复用，重复造轮子
- **协作成本**：团队成员重复回答相同的技术问题
- **AI 效能折损**：Claude 的回答缺乏专业性，需要多轮追问
- **机会成本**：无法充分利用 AI 能力，降低开发效率

**量化估算**
- 假设每天使用 Claude 10 次辅助开发
- 每次节省 5 分钟上下文准备时间 = 每天节省 50 分钟
- 每月节省约 16 小时，相当于 2 个工作日

## 2. 目标与成功指标

### 2.1 业务目标

**短期目标（MVP 发布后 1 个月）**
1. 完成核心功能开发，支持本地笔记管理和 RAG 查询
2. 验证技术方案可行性（ChromaDB + Playwright + Ollama）
3. 个人使用验证，收集使用反馈

**中期目标（3 个月内）**
1. 完善 Web 界面，提升用户体验
2. 优化查询准确率（目标：相关性 > 80%）
3. 扩展数据源支持（更多文档平台）

**长期目标（6-12 个月）**
1. 支持团队协作共享知识库
2. 提供云端同步选项（可选）
3. 构建开发者社区和插件生态

### 2.2 成功指标

**产品质量指标**
- **查询准确率**：Top-3 结果相关性 ≥ 80%
- **查询响应时间**：本地查询 < 1 秒，含 LLM 生成 < 5 秒
- **系统稳定性**：连续运行 24 小时无崩溃
- **资源占用**：内存 < 500MB，CPU 空闲 < 5%

**用户体验指标**
- **启动时间**：CLI 命令到 Web 服务可用 < 3 秒
- **操作便捷性**：添加新文档 ≤ 3 步操作
- **界面易用性**：核心功能无需文档即可上手

**业务价值指标**
- **使用频率**：日均查询次数 ≥ 10 次（个人场景）
- **知识库规模**：文档数量 ≥ 100 篇，向量条目 ≥ 1000 条
- **AI 效能提升**：Claude 对话轮次减少 30%（因上下文更充分）

**技术可行性指标**
- **向量化成功率**：≥ 99%（文档解析和向量化无失败）
- **爬虫成功率**：≥ 95%（支持主流文档平台）
- **MCP 集成稳定性**：连续调用 100 次无错误

## 3. 功能需求

### 3.1 用户故事

#### US-1: 本地笔记管理
**作为** 开发者
**我想要** 统一管理我的本地技术笔记
**以便** 快速检索和向 Claude 提供专业上下文

**验收标准**
- 支持导入本地 Markdown 文件（单个或批量）
- 支持 Obsidian 笔记库导入（保持目录结构）
- 支持 VS Code 笔记文件监控（自动同步）
- 提供笔记标签和元数据管理
- 支持全文搜索和标签过滤

#### US-2: 网页内容爬取
**作为** 开发者
**我想要** 爬取公司内部技术文档页面
**以便** 将非本地文档纳入知识库

**验收标准**
- 支持 Playwright 爬取动态网页（JavaScript 渲染）
- 支持认证登录（Cookie/Token 注入）
- 支持批量 URL 导入（CSV 或文本文件）
- 自动提取主要内容（去除导航、广告）
- 支持增量更新（检测内容变化）

#### US-3: 智能向量化存储
**作为** 开发者
**我想要** 自动将文档向量化并存储
**以便** 实现语义搜索和智能检索

**验收标准**
- 使用 Ollama 本地模型向量化（支持 nomic-embed-text）
- 自动切分长文档（chunk_size=1000, overlap=200）
- 使用 ChromaDB 持久化存储向量
- 支持元数据过滤（按标签、来源、时间）
- 向量化进度可视化（Web 界面显示）

#### US-4: 语义搜索查询
**作为** 开发者
**我想要** 用自然语言查询我的知识库
**以便** 快速找到相关技术文档和笔记

**验收标准**
- 支持 Web 界面搜索框输入查询
- 返回 Top-K 相关文档（默认 K=3，可调整）
- 显示相关度分数和匹配片段高亮
- 支持混合检索（语义搜索 + 关键词匹配）
- 查询历史记录

#### US-5: Claude MCP 集成
**作为** 开发者
**我想要** 通过 Claude Code CLI 直接查询我的知识库
**以便** 在对话中自动获取专业上下文

**验收标准**
- 实现 MCP Server 接口（stdio 通信）
- 提供 `search_knowledge_base` 工具
- Claude 可自动调用 MCP 工具获取上下文
- 支持流式返回查询结果
- 错误处理和超时保护

#### US-6: Web 管理界面
**作为** 开发者
**我想要** 通过 Web 界面管理知识库
**以便** 可视化监控和操作

**验收标准**
- 使用 Vue 3 + Vite 构建（响应式、组件化）
- 使用 Naive UI 组件库（Naive UI + Naive UI 的后台管理组件）
- 功能模块：
  - 文档列表（支持删除、重新向量化）
  - 搜索查询界面
  - 向量化进度监控
  - 系统状态（内存使用、文档数量）
  - MCP 连接状态
- 响应式设计（支持移动端访问）

## 4. 非功能需求

### 4.1 性能要求

**响应时间**
- CLI 启动到服务可用：≤ 3 秒
- 语义搜索查询（不含 LLM）：≤ 1 秒
- 含 LLM 生成回答：≤ 5 秒
- 文档向量化：≥ 10 文档/秒

**吞吐量**
- 支持并发查询：10 QPS
- 支持文档导入：100 文档/批次

**资源占用**
- 内存使用：≤ 500MB（空闲状态）
- CPU 使用：≤ 5%（空闲状态）
- 磁盘空间：≤ 1GB（1000 个文档，含向量索引）

**可扩展性**
- 支持文档数量：10,000+ 文档
- 支持向量维度：适配主流 Embedding 模型（384-1536 维）

### 4.2 安全要求

**数据隐私**
- 所有数据存储在本地，不上传云端
- 不收集任何用户使用数据
- 不包含遥测功能

**访问控制**
- Web 服务仅监听 127.0.0.1（本地访问）
- 可选配置远程访问（用户自行承担风险）
- MCP 通信使用本地 stdio（无网络暴露）

**数据保护**
- 支持数据库加密（可选，用户配置密钥）
- 敏感信息脱敏（检测密码、Token）

### 4.3 可用性要求

**可靠性**
- 系统可用性：99%（个人场景，允许偶尔重启）
- 数据持久化：ChromaDB 自动保存，不丢失数据
- 错误恢复：异常自动重启，保留运行日志

**可维护性**
- 日志记录：所有操作记录到 logs/ 目录
- 配置管理：YAML 配置文件，支持热重载
- 数据备份：支持导出向量数据库

**可测试性**
- 提供 API 测试接口
- 提供数据导入导出功能
- 支持测试模式（Mock LLM）

### 4.4 兼容性要求

**运行环境**
- Node.js：v18.17.0+（推荐 v20 LTS）
- 操作系统：macOS、Linux、WSL2（Windows 通过 WSL2）
- 架构：x64、ARM64（Apple Silicon）

**依赖服务**
- Ollama：v0.1.0+（本地 LLM 和 Embedding）
- 浏览器：Chrome 90+、Firefox 88+、Safari 14+

**数据格式**
- 支持：Markdown (.md)、纯文本 (.txt)
- 未来支持：PDF、Word、HTML

### 4.5 可观测性要求

**日志记录**
- 操作日志：文档导入、删除、查询
- 错误日志：异常堆栈、失败原因
- 性能日志：查询耗时、向量化速度

**监控指标**
- Web 界面显示：
  - 文档总数、向量总数

  - 最近查询记录

## 5. 技术方案概述

### 5.1 推荐技术栈

**后端框架**
- **Hono**：轻量级 Web 框架，类似 Express 但更快速
  - 优势：TypeScript 原生支持、性能优秀、API 简洁
  - 用途：REST API、Web 服务、MCP Server

**向量数据库**
- **ChromaDB**：本地向量数据库（JavaScript 客户端）
  - 优势：轻量级、无需单独服务、支持持久化
  - 用途：存储文档向量、语义搜索

**爬虫引擎**
- **Playwright**：浏览器自动化工具
  - 优势：支持动态网页、多浏览器、易用性强
  - 用途：爬取需要 JavaScript 渲染的文档页面

**LLM 和 Embedding**
- **Ollama**：本地 LLM 运行时
  - 优势：完全本地运行、隐私保护、免费
  - 模型选择：
    - Embedding：nomic-embed-text (384 维)
    - LLM（可选）：llama2、qwen、deepseek-coder

**前端**
- **Vue 3 + Vite**
  - 优势：响应式、组件化、开发体验好
  - UI 组件库：Naive UI（https://www.naiveui.com/zh-CN/os-theme）
  - 特点：TypeScript 支持、Tree-shaking、按需引入

### 5.2 技术架构

```
┌─────────────────────────────────────────────────────────┐
│                      用户界面层                          │
├──────────────┬──────────────┬───────────────────────────┤
│  CLI 命令    │  Web 界面    │  Claude Code (MCP)        │
│  (Terminal)  │  (浏览器)    │  (stdio 通信)             │
└──────┬───────┴──────┬───────┴───────────┬───────────────┘
       │              │                   │
       ▼              ▼                   ▼
┌─────────────────────────────────────────────────────────┐
│                    应用服务层 (Hono)                     │
├──────────┬──────────┬──────────┬──────────┬────────────┤
│ 文档导入 │ 网页爬取 │ 向量化   │ 语义搜索 │ MCP Server │
└──────┬───┴──────┬───┴──────┬───┴──────┬───┴────────────┘
       │          │          │          │
       ▼          ▼          ▼          ▼
┌─────────────────────────────────────────────────────────┐
│                    数据存储层                            │
├──────────────┬──────────────┬───────────────────────────┤
│ ChromaDB     │ 本地文件     │ Ollama (本地 LLM)         │
│ (向量存储)   │ (Markdown)   │ (Embedding + 生成)        │
└──────────────┴──────────────┴───────────────────────────┘
```

**数据流**
1. **文档导入流程**
   ```
   用户上传/导入 → 文档解析 → 文本切分 → Ollama 向量化 → ChromaDB 存储
   ```

2. **查询流程**
   ```
   用户查询 → Ollama 向量化 → ChromaDB 相似度搜索 → Top-K 结果 → 返回
   ```

3. **MCP 集成流程**
   ```
   Claude 提问 → MCP Server 接收 → 知识库查询 → 结果返回 → Claude 生成回答
   ```

### 5.3 关键技术决策

**K1: 为什么选择 ChromaDB？**
- ✅ 轻量级，无需单独部署服务
- ✅ 支持 JavaScript 客户端，与 Node.js 技术栈匹配
- ✅ 持久化存储，重启不丢失数据
- ✅ 元数据过滤，支持高级查询
- ❌ 性能不如 Milvus、Qdrant（但对个人场景足够）
- ❌ 不支持分布式（MVP 不需要）

**K2: 为什么选择 Ollama 而非 OpenAI API？**
- ✅ 完全本地运行，数据隐私保护
- ✅ 免费使用，无 API 调用成本
- ✅ 支持 CPU 和 GPU（Apple Silicon 优化）
- ❌ 模型质量略低于 GPT-4（对 Embedding 影响较小）
- ❌ 需要手动安装和配置

**K3: 为什么使用 Playwright 而非 Puppeteer？**
- ✅ 支持多浏览器（Chrome、Firefox、Safari）
- ✅ 更好的 API 设计和文档
- ✅ 官方支持，活跃维护
- ✅ 自动等待和重试机制
- ❌ 包体积稍大（可接受）

**K4: 为什么选择 Vue 3 + Vite + Naive UI？**
- ✅ Vue 3 响应式系统，开发效率高
- ✅ Vite 快速构建，HMR 秒级更新
- ✅ Naive UI 组件丰富，主题定制灵活
- ✅ TypeScript 原生支持，类型安全
- ❌ 需要构建步骤（Vite 已足够轻量）

|--------|------|------|---------|
| Ollama 向量化质量不足 | 高 | 中 | 提供切换 OpenAI API 选项 |
| ChromaDB 性能瓶颈（文档过多） | 中 | 低 | 优化索引、分片存储 |
| Playwright 爬虫被反爬 | 中 | 中 | 模拟用户行为、添加延迟 |
| MCP 协议变更 | 高 | 低 | 版本锁定、兼容性测试 |
| 本地 LLM 生成速度慢 | 中 | 高 | 可选流式输出、缓存结果 |

### 6.2 产品风险

| 风险项 | 影响 | 概率 | 缓解措施 |
|--------|------|------|---------|
| 用户不会配置 Ollama | 高 | 高 | 提供详细安装文档、自动检测 |
| 查询准确率不达预期 | 高 | 中 | 优化 chunk 策略、调整相似度阈值 |
| Web 界面体验差 | 中 | 中 | 迭代优化、收集用户反馈 |
| 用户使用门槛高 | 高 | 高 | 简化 CLI 命令、提供配置向导 |

### 6.3 依赖风险

**外部依赖**
- **Ollama 服务**：必须提前安装，否则无法向量化
  - 缓解：启动时检测，提供安装指引
- **Node.js 生态**：依赖 npm 包稳定性
  - 缓解：锁定版本、定期更新

**数据依赖**
- **本地文档**：用户需要有本地 Markdown 文件
  - 缓解：提供示例数据、支持空库启动
- **文档访问权限**：爬取内网文档需要认证
  - 缓解：支持 Cookie/Token 配置

### 6.4 合规风险

| 风险项 | 说明 | 缓解措施 |
|--------|------|---------|
| 知识产权 | 爬取公司文档可能违规 | 明确免责声明、仅限个人使用 |
| 数据隐私 | 本地存储敏感信息 | 支持加密、不上传云端 |
| 开源协议 | 依赖开源组件需合规 | 使用 MIT/Apache 协议库 |
