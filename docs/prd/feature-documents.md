## 1. 问题陈述

### 1.1 核心痛点

**背景场景**
- 用户当前依赖导入外部 Markdown 文件或爬取网页内容来构建知识库
- 缺乏内置的笔记编辑能力，用户需要在其他工具（如 Obsidian、VS Code）中编辑后再导入
- 编辑-导入-向量化流程割裂，用户体验不连贯
- 无法快速记录临时想法和灵感

**具体痛点**
1. **内容创建效率低**
   - 需要在外部工具编辑 → 保存 → 手动导入 → 向量化
   - 流程繁琐，每次创建新笔记需要多个步骤
   - 无法快速记录灵感和临时笔记

2. **内容管理分散**
   - 笔记分散在多个工具和平台
   - 缺乏统一的笔记管理和组织方式
   - 无法在单一界面完成创建、编辑、搜索全流程

3. **知识闭环不完整**
   - 系统只能消费已有内容，无法创造新内容
   - 从"知识检索"到"知识创造"的闭环缺失
   - 限制了系统作为个人知识管理平台的潜力

4. **协作能力不足**
   - 无法在 Web 界面直接编辑笔记
   - 缺乏富文本编辑和格式化能力
   - 无法快速调整和优化已导入的内容

### 1.2 目标用户

**主要用户画像**
- **角色**：软件开发工程师、知识工作者
- **技术背景**：熟悉 Markdown 语法，习惯使用笔记工具
- **工作场景**：
  - 需要快速记录技术笔记和灵感
  - 需要整理和归档外部文档内容
  - 需要在知识库中直接编辑和优化内容
- **痛点**：编辑-导入流程割裂，缺乏统一的笔记创作环境

**用户规模**
- 初期目标：个人用户，作为现有知识库的补充
- 未来扩展：支持团队协作笔记共享

### 1.3 不解决的成本

**如果不实现本地笔记功能**
- **效率损失**：每次创建新笔记需要 3-5 分钟（外部编辑 + 导入），直接编辑只需 30 秒
- **体验割裂**：用户需要在多个工具间切换，认知负担增加
- **使用门槛**：新用户需要先学习如何导入内容，无法直接上手使用
- **功能局限**：系统仅能"消费"知识，无法"创造"知识，限制平台潜力
- **竞争优势**：缺乏笔记编辑功能，相比 Obsidian + RAG 插件等竞品缺少核心竞争力

**量化估算**
- 假设每天需要创建 5 条新笔记
- 每次节省 3 分钟流程时间 = 每天节省 15 分钟
- 每月节省约 7.5 小时，相当于 1 个工作日

### 3.2 标签功能详细设计

#### 标签功能概述

标签是笔记组织的核心功能，允许用户为笔记添加关键词标签，便于快速筛选和检索。

#### 标签功能需求

**T-1: 为笔记添加标签**
- 在编辑器界面提供标签输入框
- 支持输入多个标签，用逗号或回车分隔
- 标签创建后自动保存
- 支持从已有标签中选择

**T-2: 标签筛选**
- 在笔记列表页面显示所有标签
- 点击标签可筛选出包含该标签的笔记
- 支持多标签组合筛选（AND 逻辑）
- 筛选结果实时更新

**T-3: 标签管理**
- 查看所有标签列表
- 显示每个标签关联的笔记数量
- 重命名标签（自动更新所有关联笔记）
- 删除标签（从所有笔记中移除）
- 标签按使用频率排序

**T-4: MCP 标签查询**
- MCP API 支持按标签查询笔记
- 支持获取所有标签列表
- 支持标签自动补全
- 支持多标签组合查询

#### 标签数据结构

```typescript
interface Tag {
  id: string;          // 标签唯一 ID
  name: string;        // 标签名称
  color?: string;      // 标签颜色（v1.1）
  createdAt: Date;     // 创建时间
}

interface Note {
  // ... 其他字段
  tags: string[];      // 标签 ID 列表
}
```

#### MCP 标签查询 API 示例

```typescript
// 获取所有标签
mcp.list_tags()

// 按标签查询笔记
mcp.query_notes_by_tags({ tags: ["javascript", "tutorial"] })

// 标签自动补全
mcp.suggest_tags({ prefix: "jav" })
```

## 4. 非功能需求

- 向量化触发：保存后 ≤ 1 秒

**资源占用**
- 编辑器内存：≤ 50MB（单个实例）
- 编辑器包体积：≤ 200KB（gzipped）
- 笔记存储：≤ 1MB/篇（纯文本）

**并发能力**
- 支持同时编辑多个笔记（多标签页）

**数据保护**
- 编辑内容暂存到 LocalStorage（加密）
- 防止 XSS 攻击（Markdown 渲染过滤）

- 编辑功能需要登录认证（未来版本）
- 删除操作需要二次确认

- 所有笔记存储在本地
- 不上传任何内容到云端
- 不收集编辑行为数据

### 4.3 可用性要求

**可靠性**
- 自动保存成功率：≥ 99%
- 编辑器崩溃率：< 0.1%
- 数据丢失率：0%（LocalStorage + 服务器双保险）

**可维护性**
- 编辑器组件化，易于升级
- 提供编辑器配置接口
- 支持插件扩展（未来版本）

**可测试性**
- 提供编辑器单元测试

### 4.4 兼容性要求

**浏览器兼容**

- Safari 14+
- Edge 90+

**移动端兼容**

**数据格式**

### 4.5 用户体验要求

## 5. 技术方案概述

### 5.1 推荐技术栈

**Markdown 编辑器选型**

经过调研，推荐以下三种方案，按优先级排序：

#### 方案 1：md-editor-v3 ⭐（推荐）

**简介**
- GitHub：https://github.com/imzbf/md-editor-v3
- 文档：https://imzbf.github.io/md-editor-v3/zh-CN/index
- 基于 Vue 3 + TSX 开发的 Markdown 编辑器

**优势**
- ✅ 专为 Vue 3 设计，TypeScript 友好
- ✅ 功能完善，支持实时预览、语法高亮、数学公式、Mermaid 图表
- ✅ 内置多种主题（深色/浅色）
- ✅ 支持快捷键和工具栏
- ✅ 社区活跃，文档详细（中文）
- ✅ 包体积适中（~150KB gzipped）
- ✅ 支持图片粘贴和上传
- ✅ 支持表格、代码高亮、数学公式（注：表格、公式功能在 MVP 阶段不启用）

**劣势**
- ❌ 相对较新，生态不如 TipTap 成熟
- ❌ 定制化能力略逊于 TipTap

**适用场景**
- 需要 Vue 3 原生集成的项目
- 需要快速上手的 Markdown 编辑器
- 对中文文档支持有要求

**评分**：9/10

#### 方案 2：TipTap

**简介**
- 官网：https://tiptap.dev/
- 基于 ProseMirror 的无头富文本编辑器框架

**优势**
- ✅ 架构优秀，基于 ProseMirror（Notion、GitLab 都在用）
- ✅ 高度可定制，支持扩展开发
- ✅ 性能优秀，支持大文档编辑
- ✅ 支持协作编辑（Y.js 集成）
- ✅ 社区活跃，生态成熟
- ✅ TypeScript 友好

**劣势**
- ❌ 学习曲线陡峭，配置复杂
- ❌ 需要自行构建 Markdown 支持
- ❌ 包体积较大（需要配置 Tree-shaking）
- ❌ 快速上手成本高

**适用场景**
- 需要高度定制化的编辑器
- 未来需要协作编辑功能
- 团队有足够的技术能力

**评分**：8/10（MVP 阶段略重）

#### 方案 3：Milkdown

**简介**

### 5.2 技术架构

**前端架构**

```
┌─────────────────────────────────────────────────────────┐
│                      Vue 3 应用层                        │
├──────────────┬──────────────┬───────────────────────────┤
│  路由层      │  状态管理    │  组件层                    │
│  Vue Router  │  (可选)      │  Naive UI + 自定义组件   │
└──────┬───────┴──────┬───────┴───────────┬───────────────┘
       │              │                   │
       ▼              ▼                   ▼
┌─────────────────────────────────────────────────────────┐
│                    编辑器层                              │
├──────────────┬──────────────┬───────────────────────────┤
│  md-editor-  │  自动保存    │  图片上传                  │
│  v3 组件     │  (30s 定时)  │  (未来版本)               │
└──────┬───────┴──────┬───────┴───────────┬───────────────┘
       │              │                   │
       ▼              ▼                   ▼
┌─────────────────────────────────────────────────────────┐
│                    API 层 (Axios)                       │
├──────────┬──────────┬──────────┬──────────┬────────────┤
│  创建笔记 │  更新笔记 │  删除笔记 │  查询笔记 │  向量化   │
└──────┬───┴──────┬───┴──────┬───┴──────┬───┴────────────┘
       │          │          │          │
       ▼          ▼          ▼          ▼
┌─────────────────────────────────────────────────────────┐
│                  后端服务层 (Hono)                       │
└─────────────────────────────────────────────────────────┘
```

**数据流**

1. **创建笔记流程**
   ```
   用户输入 → 编辑器 → 点击保存 → API 请求 → 保存到文件系统 → 触发向量化 → 返回成功 → 更新 UI
   ```

2. **自动保存流程**
   ```
   用户输入 → 30秒定时器 → API 请求 → 保存草稿 → 更新 UI 状态
   ```

3. **向量化流程**
   ```
   保存成功 → 消息队列 → 向量化服务 → ChromaDB 存储 → 更新搜索索引
   ```

**存储架构设计**

```
┌─────────────────────────────────────────────────────────────────────┐
│                         本地笔记存储架构                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    文件系统存储层                            │   │
│  │  .devrag/notes/                                              │   │
│  │  ├── 2024-03-27-note1.md       ← 完整 Markdown 内容         │   │
│  │  ├── 2024-03-27-note2.md                                    │   │
│  │  └── ...                                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                         ↓ 文件路径关联                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    ChromaDB 向量存储层                       │   │
│  │  {                                                            │   │
│  │    id: "chunk_001",                                          │   │
│  │    embedding: [0.23, -0.45, ...],  ← 向量数据               │   │
│  │    content: "这是 chunk 内容...",    ← 原始文本              │   │
│  │    metadata: {                                               │   │
│  │      filePath: ".devrag/notes/2024-03-27-note1.md", ← 关联  │   │
│  │      chunkIndex: 0,                                          │   │
│  │      title: "笔记标题",                                       │   │
│  │      tags: ["javascript", "tutorial"]                        │   │
│  │    }                                                          │   │
│  │  }                                                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

查询流程：
1. ChromaDB 向量搜索 → 返回匹配的 chunk（包含 filePath）
2. 通过 filePath 读取完整 Markdown 文件
3. 返回完整文档 + 高亮匹配的 chunk
```

**存储路径结构**

```
.devrag/
├── notes/                          # 笔记原始文件
│   ├── 2024-03-27-{uuid}-note1.md  # 文件命名: {日期}-{uuid}-{标题}.md
│   └── 2024-03-27-{uuid}-note2.md
├── documents-metadata.json         # 元数据索引（可选，用于快速列表查询）
└── chromadb/                       # 向量数据（Docker 卷）
```

**数据关联关系**

| 存储位置 | 存储内容 | 用途 |
|---------|---------|------|
| 文件系统 (.devrag/notes/) | 完整 Markdown 文件 | 用户编辑、版本控制、外部工具访问 |
| ChromaDB metadata | filePath、chunkIndex、tags、title | 向量搜索时的关联查询 |
| ChromaDB document | chunk 原始文本 | 搜索结果片段展示 |
| ChromaDB embedding | 向量数据 | 语义相似度计算 |

**核心优势**
- ✅ **简单高效**：无需额外数据库，通过文件路径天然关联
- ✅ **可移植性**：笔记文件可直接用其他工具编辑（VS Code、Obsidian）
- ✅ **版本控制友好**：Markdown 文件可纳入 Git 管理
- ✅ **备份方便**：直接复制 `.devrag/notes/` 目录即可
- ✅ **查询高效**：向量搜索返回 filePath，直接读取文件

**注意事项**
- ⚠️ 文件删除/移动时需要更新 ChromaDB 索引
- ⚠️ 并发写入需要文件锁机制（单用户场景可忽略）
- ⚠️ 大文件读取性能需要优化（可考虑分块缓存）

### 5.3 关键技术决策

**K1: 为什么选择 md-editor-v3 而非 TipTap？**
- ✅ Vue 3 原生支持，无需额外配置
- ✅ 开箱即用，快速集成
- ✅ 中文文档，降低学习成本
- ✅ 功能完善，满足 MVP 需求
- ❌ TipTap 定制化能力强，但 MVP 阶段不需要
- ❌ TipTap 学习曲线陡峭，开发成本高

**K2: 为什么选择自动保存而非手动保存？**
- ✅ 防止数据丢失，提升用户体验
- ✅ 符合现代编辑器习惯（Google Docs、Notion）
- ✅ 降低用户操作负担
- ❌ 需要处理并发保存冲突（通过去重解决）

**K3: 为什么向量化在后台异步执行？**
- ✅ 不阻塞用户编辑操作
- ✅ 提升保存响应速度
- ✅ 支持向量化失败重试
- ❌ 搜索可能有短暂延迟（可接受）

**K4: 为什么使用 LocalStorage 存储草稿？**
- ✅ 防止浏览器关闭导致数据丢失
- ✅ 双重保险（LocalStorage + 服务器）
- ✅ 离线时可恢复草稿
- ❌ 需要处理存储空间限制（5MB 限制）

**K5: 为什么选择文件路径关联而非额外数据库？**
- ✅ **架构简单**：无需引入 SQLite/PostgreSQL，降低系统复杂度
- ✅ **原生可编辑**：笔记文件可用 VS Code、Obsidian 等工具直接编辑
- ✅ **版本控制友好**：Markdown 文件可纳入 Git，天然支持版本历史
- ✅ **备份方便**：直接复制 `.devrag/notes/` 目录即可迁移
- ✅ **符合本地工具定位**：local-rag 是本地优先工具，文件存储更契合
- ❌ 需要处理文件删除时的索引更新（通过文件监听解决）
- ❌ 大文件读取性能略低于数据库（可通过缓存优化）

**对比方案：引入 SQLite 数据库**
| 维度 | 文件路径关联 | SQLite 数据库 |
|------|-------------|--------------|
| 系统复杂度 | 低（无额外依赖） | 中（需数据库驱动） |
| 可编辑性 | 高（任何文本编辑器） | 低（需通过应用） |
| 版本控制 | Git 友好 | 需要导出功能 |
| 查询性能 | 中（需读文件） | 高（索引优化） |
| 备份迁移 | 复制目录即可 | 需要数据库导出 |
| 适用场景 | 本地单用户工具 | 多用户/协作场景 |

**结论**：对于 local-rag 这种本地单用户知识库工具，文件路径关联方案更符合产品定位，后续如需支持多用户或协作功能，可考虑迁移到 SQLite。
