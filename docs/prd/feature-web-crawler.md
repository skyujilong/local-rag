## 1. 问题陈述

### 1.1 核心痛点

**背景场景**
- 系统已具备基础的网页爬取能力（`POST /api/crawl`），但仅支持单页面爬取，且缺乏前端交互界面
- 用户需要将在线文档、技术博客、API 文档等网页内容导入知识库，当前操作流程繁琐
- 爬取的内容缺少精细化的清洗和分块控制，搜索质量受限
- 爬取结果与 Documents 功能未实现深度集成，无法在笔记列表中管理爬取内容

**具体痛点**
1. **单页面爬取能力不足**
   - 仅支持输入单个 URL 爬取，无法批量处理
   - 不支持站点地图（Sitemap）解析和递归爬取
   - 无法爬取结构化的多页文档站点（如 VuePress、Docusaurus）

2. **缺少前端交互界面**
   - 当前只能通过 API 调用爬取，普通用户无法使用
   - 无法预览爬取内容、调整参数、查看进度
   - 缺少爬取历史记录管理

3. **内容清洗不彻底**
   - 当前仅移除 nav/header/footer 等基础元素
   - 无法精确提取文章主体内容，常混入无关文本
   - 不支持对爬取内容进行二次编辑和优化

4. **无法爬取需要登录的页面**
   - 许多高质量内容（如内部文档、付费课程、私有 GitLab Wiki）需要登录后才能访问
   - 当前爬虫无法处理登录流程，遇到登录页直接失败
   - 缺少 Cookie/Session 管理和登录状态维持能力
   - 用户无法手动登录后复用认证状态进行爬取

5. **与 Documents 功能割裂**
   - 爬取结果直接作为文本文档导入，丢失网页元信息（URL、来源站点等）
   - 无法区分手动创建的笔记和爬取导入的文档
   - 缺少来源 URL 追溯能力

### 1.2 目标用户

**主要用户画像**
- **角色**：软件开发工程师、技术知识工作者
- **技术背景**：需要将在线技术文档、博客文章、API 文档等导入本地知识库
- **工作场景**：
  - 爬取官方技术文档（如 React、Vue、Rust 文档站点）供离线语义搜索
  - 批量导入感兴趣的技术博客文章
  - 定期同步更新在线文档的本地副本
  - 收集和整理分散在多个网站的知识内容
- **痛点**：手动复制粘贴效率低，现有爬取工具与知识库不打通

**用户规模**
- 初期目标：现有 devrag-cli 用户，作为知识导入的核心方式之一
- 未来扩展：面向需要定期爬取和同步在线内容的团队用户

### 1.3 不解决的成本

**如果不实现增强版 Web Crawler 功能**
- **效率损失**：用户需手动复制粘贴网页内容，每篇文章约需 2-5 分钟；自动爬取仅需 5-10 秒
- **内容覆盖不足**：无法批量导入整个文档站点，知识库覆盖面受限
- **内容质量低**：手动导入的内容未经清洗，包含导航、广告等无关信息，降低搜索质量
- **功能闭环缺失**：Documents 功能只能手动创建或文件导入，缺少在线内容自动获取能力
- **竞争优势弱化**：相比支持网页导入的知识库工具（如 Obsidian Web Clipper），功能差距明显

**量化估算**
- 假设用户每周需导入 10 篇网页内容
- 手动操作每篇 3 分钟 vs 自动爬取 10 秒 = 每周节省约 28 分钟
- 批量爬取文档站点（如 100 页）：手动操作 5 小时 vs 自动爬取 10 分钟

## 2. 目标与成功指标

### 2.1 业务目标

**短期目标（MVP 发布后 1 个月）**
1. 提供完整的 Web 爬取前端界面，支持单页面 URL 爬取
2. 支持站点地图（Sitemap）解析和批量爬取
3. 实现内容智能清洗，去除无关元素
4. 与 Documents 功能深度集成，爬取内容作为文档导入
5. 支持需认证页面的爬取（Cookie 注入、浏览器交互式登录、HTTP Header 注入）

**中期目标（3 个月内）**
1. 支持递归爬取模式，自动发现和爬取子页面
2. 支持定时爬取和内容更新检测
3. 优化内容提取算法，提升文档质量
4. 认证自动续期与多账户管理

**长期目标（6-12 个月）**
1. 支持自定义爬取规则（CSS 选择器、正则表达式）
2. 支持爬取结果对比和版本管理
3. 集成 AI 内容摘要和自动标签生成
4. 支持多种导出格式（Markdown、PDF、EPUB）

### 2.2 成功指标

**产品质量指标**
- **爬取成功率**：>= 95%（针对正常可访问的公开网页）
- **认证页面爬取成功率**：>= 90%（Cookie/Header 注入模式，受第三方认证策略影响，指标适当放宽）
- **浏览器登录完成率**：>= 80%（用户发起浏览器登录后成功提取到认证状态的比例，未完成原因含用户放弃、超时等）
- **内容清洗准确率**：>= 90%（有效内容占比）
- **向量化成功率**：>= 99%（爬取内容导入后成功向量化）
- **搜索相关性**：爬取导入的内容可通过语义搜索准确找到

**用户体验指标**
- **单页面爬取时间**：< 15 秒（含内容提取和清洗）
- **批量爬取吞吐**：>= 5 页/分钟
- **操作便捷性**：从输入 URL 到完成导入 <= 3 步操作
- **进度可见性**：批量爬取实时显示进度和状态

**业务价值指标**
- **使用频率**：>= 30% 用户使用过网页爬取功能
- **导入量占比**：爬取导入占所有文档导入量的 >= 20%
- **内容留存率**：爬取导入的内容 7 天后仍存在 >= 70%

## 3. 功能需求

### 3.1 用户故事

#### US-1: 单页面爬取
**作为** 开发者
**我想要** 输入一个 URL 并爬取该网页内容
**以便** 快速将感兴趣的网页文章导入知识库

**验收标准**
- 提供 URL 输入框和"爬取"按钮
- 输入合法 URL 后点击爬取，系统自动获取网页内容
- 爬取完成后显示内容预览（标题、正文摘要、字数统计）
- 用户可确认导入或取消
- 导入后自动触发向量化，内容可通过语义搜索找到

#### US-2: 站点地图爬取
**作为** 开发者
**我想要** 输入站点的 Sitemap URL，批量爬取站点下的所有页面
**以便** 将整个技术文档站点导入知识库

**验收标准**
- 支持输入 Sitemap XML 的 URL（如 `https://example.com/sitemap.xml`）
- 自动解析 Sitemap 中的所有 URL
- 显示解析结果（URL 总数、预计耗时）
- 用户可选择全部爬取或勾选部分页面
- 批量爬取过程中实时显示进度（已完成/总数、当前 URL、成功/失败数）
- 支持暂停和继续爬取

#### US-3: 递归爬取
**作为** 开发者
**我想要** 输入一个起始 URL，系统自动发现并爬取同域名下的子页面
**以便** 爬取没有 Sitemap 的文档站点

**验收标准**
- 支持设置递归深度（1-3 级，默认 2 级）
- 支持设置 URL 过滤规则（包含/排除模式）
- 仅爬取同域名下的页面，不跨域
- 显示发现的 URL 列表，用户可确认后开始爬取
- 爬取过程中实时显示进度

#### US-4: 内容预览和编辑
**作为** 开发者
**我想要** 在导入前预览爬取到的内容
**以便** 确认内容质量并在必要时进行编辑

**验收标准**
- 爬取完成后显示内容预览面板
- 预览内容包括：标题、正文内容、字数、来源 URL
- 支持在预览状态下编辑标题和内容
- 支持为导入内容添加标签
- 编辑后的内容用于后续向量化

#### US-4b: 批量导入确认流程
**作为** 开发者
**我想要** 在批量爬取完成后，统一确认导入的内容
**以便** 控制导入质量，跳过不满意的页面

**验收标准**
- 批量爬取完成后进入"导入确认"界面，显示所有爬取结果的总览列表
- 总览列表每行显示：标题（可点击展开预览）、字数、来源域名、清洗质量评分（高/中/低，基于正文占比）
- 默认全选所有结果，用户可取消勾选不需要的页面
- 标注已导入过的 URL（灰色标记，默认取消勾选），用户可勾选覆盖（详见 US-7 重复导入检测）
- 提供"全选 / 全选高质量 / 取消全选"快捷操作
- 支持"批量设置标签"功能（为本次导入的所有页面统一添加标签）
- 点击"导入 N 个页面"后，系统批量创建文档并触发向量化
- 导入过程中显示进度（已导入 X/总 N），完成后显示汇总（成功数、失败数、跳过数）

#### US-5: 内容清洗
**作为** 开发者
**我想要** 爬取的内容自动去除导航、广告等无关信息
**以便** 导入的知识库内容干净、精准，提高搜索质量

**验收标准**
- 自动去除导航栏、页脚、侧边栏、广告区域
- 自动去除 Cookie 提示、弹窗等干扰元素
- 保留文章主体内容（标题、段落、代码块、列表等）
- 保留有意义的结构化内容（表格、引用等）
- 清洗后的内容格式为干净的纯文本或 Markdown

#### US-6: 爬取任务管理
**作为** 开发者
**我想要** 查看和管理我的爬取任务历史
**以便** 了解已导入的网页内容和爬取状态

**验收标准**
- 显示爬取任务列表（URL、标题、状态、时间）
- 支持按状态筛选（成功、失败、进行中）
- 支持重新爬取失败的任务
- 显示每个任务的详细信息（内容摘要、错误原因等）

#### US-7: 与 Documents 功能集成
**作为** 开发者
**我想要** 爬取的网页内容与手动创建的笔记统一管理
**以便** 在同一个知识库中搜索和管理所有内容

**验收标准**
- 爬取导入的文档出现在 Documents 笔记列表中
- 文档自动标记来源（`source: "web"`）和原始 URL
- 支持为爬取内容设置标签（自动生成 + 手动添加）
- 支持在笔记详情中查看原始网页链接
- 删除笔记时同步清理向量化数据
- **重复导入检测**：
  - 导入时以 URL 为唯一键检测重复（精确匹配，忽略 hash 和 query 参数中的追踪参数如 `utm_*`）
  - 单页面爬取：若该 URL 已导入过，提示"该页面已于 {date} 导入过"，提供"覆盖更新"和"仍然导入（创建副本）"两个选项
  - 批量爬取：导入确认界面自动标注已导入的 URL（灰色标记），默认跳过已导入页面，用户可手动勾选覆盖
  - 覆盖更新：保留原文档 ID，更新内容和向量化数据，保留用户手动编辑的标签

#### US-8: 爬取配置
**作为** 开发者
**我想要** 自定义爬取参数
**以便** 针对不同网站调整爬取策略

**验收标准**
- 支持设置超时时间（默认 30 秒，最大 120 秒）
- 支持设置请求间隔（默认 1 秒，防止对目标站点造成压力，批量模式下最小 1 秒不可调小）
- 支持设置 User-Agent
- 支持指定 CSS 选择器提取特定区域内容

> **并发数说明**：并发数不再作为用户可配置项。系统根据场景自动管理：
> - 单页面爬取：每次请求独立使用浏览器实例，无需用户关心并发
> - 批量爬取：默认串行（并发 1），系统强制限制同时活跃浏览器实例 <= 2（详见 4.1 批量爬取性能约束）
> - 此设计避免用户误设高并发导致系统卡死，详见 K3 技术决策

#### US-9: 系统登录认证（WC-12）
**作为** 开发者
**我想要** 爬取需要登录才能访问的网页内容
**以便** 将内部文档、付费课程、私有 Wiki 等需要认证的内容导入知识库

**验收标准**
- 支持三种认证模式：
  1. **Cookie 注入**（P0）：用户手动粘贴 Cookie 字符串，爬虫自动携带
  2. **浏览器登录**（P1）：在服务端启动 Playwright 有头浏览器窗口，用户在本地浏览器窗口中手动完成登录
  3. **HTTP Header 注入**（P1）：用户手动设置 Authorization Header（支持 Bearer Token、Basic Auth 等）
- **浏览器登录交互流程**：
  1. 前端调用 `POST /api/crawl/auth/launch-browser`，传入登录页 URL
  2. 服务端启动 Playwright 有头浏览器（`headless: false`），导航到登录页
  3. 服务端通过 SSE（Server-Sent Events）推送状态 `browser_launched`
  4. 用户在弹出的本地浏览器窗口中手动完成登录操作
  5. 前端展示等待状态 + "完成登录"按钮（供用户在登录完成后主动点击）
  6. 前端调用 `POST /api/crawl/auth/complete-login`，或服务端通过页面 URL 变化自动检测登录完成
  7. 服务端提取 Cookie/Session，加密存储，返回 authProfileId
- **浏览器登录环境限制**：
  - 浏览器登录要求运行环境支持 GUI 显示（本机桌面环境）
  - 以下环境**不支持**浏览器登录，仅支持 Cookie 注入和 Header 注入：
    - SSH 远程连接（无 display）
    - Docker 容器部署（无 GUI，除非配置 X11 转发）
    - CI/CD 管道
  - 检测到无 GUI 环境时，`launch-browser` API 返回错误 `{ error: "no_display", message: "当前环境不支持弹出浏览器，请使用 Cookie 注入方式" }`
  - 前端据此隐藏"浏览器登录"选项卡，仅展示 Cookie 注入和 Header 注入
- 登录成功后系统自动保存认证状态（Cookie/Token），供后续爬取复用
- 认证状态可命名保存和管理（如"公司 GitLab"、"Notion"）
- 认证状态过期检测：爬取时若返回登录页，提示用户重新认证
- **批量爬取中认证过期处理**：
  1. 检测到认证过期时，暂停当前批量任务（已完成的页面结果保留）
  2. 前端弹出提示"认证已过期，请重新登录"，提供快捷重新登录按钮
  3. 用户完成重新登录后，任务自动从暂停处继续，不重复爬取已成功的页面
  4. 若用户选择"跳过继续"，则后续页面在无认证状态下爬取（可能失败），失败页标记为 `auth_expired`
  5. 若用户选择"终止任务"，则保存已完成的结果，用户可在导入确认界面选择性导入
- 支持查看和删除已保存的认证状态
- 敏感信息（Cookie、Token）加密存储，不明文日志输出（详见 4.2 安全要求）

### 3.3 验收标准

#### 功能验收

**WC-1: URL 输入与单页面爬取**
```gherkin
Given 用户在 Web Crawler 页面
When 用户输入合法的 URL（如 https://example.com/article）并点击"爬取"
Then 系统应在 15 秒内返回爬取结果
And 显示内容预览（标题、正文、字数统计）
And 如果 URL 不可访问，显示清晰的错误信息
```

**WC-2: 内容自动清洗**
```gherkin
Given 系统成功爬取一个网页
When 执行内容清洗流程
Then 系统应移除导航栏、页脚、侧边栏、广告区域
And 提取文章主体内容（main/article 区域）
And 清洗后的内容应保留标题层级、段落结构和代码块
And 无关文本占比应低于 10%
```

**WC-4: 导入并自动向量化**
```gherkin
Given 用户在内容预览界面确认导入
When 用户点击"导入"按钮
Then 系统应将内容创建为文档（通过 Documents 功能）
And 文档元数据包含来源 URL 和爬取时间
And 系统自动触发向量化流程
And 导入完成后该内容可通过语义搜索找到
```

**WC-6: 站点地图爬取**
```gherkin
Given 用户输入一个有效的 Sitemap URL

## 4. 非功能需求

### 4.1 性能要求

**响应时间**
- 单页面爬取（含清洗）：<= 15 秒
- 内容预览加载：<= 2 秒
- 批量爬取吞吐：>= 5 页/分钟
- 爬取进度更新频率：<= 1 秒
- 认证页面首跳延迟：额外增加 <= 3 秒（注入 Cookie/Header 的时间开销）

**资源占用**
- 浏览器实例内存：<= 200MB/实例（Playwright Chromium）
- 并发爬取任务：最大 5 个
- 请求间隔：>= 1 秒（防止 DDoS 目标站点）
- 单次爬取最大内容：10MB

**并发能力**
- 支持同时运行一个批量爬取任务和多个单页面爬取
- 浏览器实例池管理，自动回收空闲实例

**批量爬取性能约束**

批量爬取场景（Sitemap/递归）下，多个页面连续爬取会持续占用 Playwright 浏览器实例和内存。为防止资源耗尽导致系统卡死，必须遵循以下约束：

| 约束项 | 限制值 | 说明 |
|--------|--------|------|
| 同时活跃浏览器实例数 | <= 2 | 每个实例约 200MB，2 个即 400MB。超过此限制时排队等待，不可创建新实例 |
| 批量爬取默认并发数 | 1（串行） | 批量模式下默认逐页串行爬取，用户可调整为 2，但不得超过活跃实例数上限 |
| 单个批量任务最大页面数 | 默认 200，硬上限 500 | 超过 200 页提示"任务较大，可能影响性能，是否继续？"。500 页为绝对上限，不可突破 |
| 每页爬取后强制 GC 间隔 | 50 页 | 每爬取 50 页后关闭当前浏览器实例、重建新实例，释放累积的内存泄漏 |
| 爬取任务总内存上限 | 1GB | 后端进程 RSS 超过 1GB 时暂停新页面爬取，等待当前任务完成后回收内存再继续 |
| 请求间隔 | >= 1 秒 | 批量模式下每页之间强制等待，默认 1 秒，用户可调大（最大 10 秒），不可调小 |

**批量爬取资源回收策略**

```
批量任务启动
  │
  ├─ 创建浏览器实例（复用实例池）
  │
  ├─ 爬取页面 [1..50]
  │    └─ 逐页串行，每页间隔 >= 1 秒
  │
  ├─ 第 50 页完成后：关闭浏览器实例 → 触发 GC → 创建新实例
  │
  ├─ 爬取页面 [51..100]
  │    └─ 同上
  │
  ├─ 第 100 页完成后：同上回收
  │
  └─ ... 直至全部完成或用户暂停
```

**内存监控与熔断**

- 后端每 10 秒检测一次进程 RSS 内存占用
- RSS > 800MB：日志告警，降低爬取速度（间隔翻倍）
- RSS > 1GB：暂停爬取，等待 30 秒让 GC 回收后恢复
- RSS 持续 > 1GB 超过 3 分钟：自动终止当前批量任务，保存已完成结果，提示用户"内存不足，任务已保存进度"
- 浏览器实例崩溃（page crash / browser disconnect）：自动重试当前页面（最多 2 次），重试失败则标记该页为 failed 并继续下一页

### 4.2 安全要求

**爬取行为合规**
- 遵守 robots.txt 协议（可选检查，默认不阻止）
- 设置合理的请求间隔（>= 1 秒），不对目标站点造成压力
- 设置 User-Agent 标识，不伪装为搜索引擎爬虫
- 单站点爬取上限：默认 200 页，用户可强制突破至 500 页（系统提示性能风险确认）

**数据安全**
- Cookie 等认证信息安全存储，不明文日志输出
- 爬取内容经 DOMPurify 过滤，防止 XSS
- 不存储敏感的网页表单数据

**认证信息加密方案**
- 使用 Node.js `crypto` 模块的 AES-256-GCM 算法加密敏感数据（Cookie、Token）
- 加密密钥基于机器唯一标识（MAC 地址 + 主机名的 SHA-256 哈希）自动生成，无需用户额外配置
- 密钥存储在本地文件系统，权限设为 `0600`（仅当前用户可读写）
- 应用重启后自动从本地文件恢复密钥，无需用户干预
- 密钥丢失场景：若机器标识发生变化（如重装系统），已加密的认证数据无法解密，用户需重新登录；系统检测到解密失败时提示"认证数据已失效，请重新配置"

**访问控制**
- 仅限本地用户使用（localhost 绑定）
- 不暴露爬取能力为公开 API

### 4.3 可用性要求

**可靠性**
- 爬取成功率：>= 95%（正常可访问页面）
- 网络超时自动重试（最多 3 次）
- 部分页面爬取失败不影响批量任务继续
- 浏览器实例崩溃后自动恢复

**可维护性**
- 爬取服务模块化，独立于 Documents 服务
- 清洗规则可配置，支持自定义选择器
- 错误信息清晰，便于定位问题

### 4.4 兼容性要求

**目标网站兼容**
- 静态 HTML 页面：100% 支持
- JavaScript 动态渲染页面：通过 Playwright 支持（需等待加载）
- SPA 应用：部分支持（依赖 waitForSelector 配置）
- 需要 Cookie 的页面：通过配置支持

**内容格式兼容**
- HTML 页面：完整支持
- XML/Sitemap：完整支持解析
- JSON API：不支持（MVP 阶段）
- PDF/文档文件：不支持（MVP 阶段）

**数据流**

1. **单页面爬取流程**
   ```
   输入 URL → 启动爬取 → Playwright 加载页面 → 内容清洗 → 预览 → 确认导入 → Documents 创建 → 向量化
   ```

2. **站点地图爬取流程**
   ```
   输入 Sitemap URL → 解析 XML → 显示 URL 列表 → 用户选择 → 批量爬取（串行 + 间隔）→ 逐页清洗 → 批量导入 → 批量向量化

   ```
   输入起始 URL → 爬取首页 → 提取同域链接 → 按深度递归 → URL 去重 → 显示发现列表 → 用户确认 → 批量爬取 → 导入
   ```

4. **浏览器登录认证流程**
   ```
   前端调用 launch-browser API → 后端启动 Playwright 有头浏览器（headless: false）→

   ```

5. **Cookie/Token 注入流程**
   ```
   用户选择认证方式（Cookie/Header）→ 粘贴认证信息 → 验证格式 →
   加密存储 → 爬取时自动注入请求头 → 认证失败时提示更新
   ```

### 5.3 关键技术决策

**K1: 为什么继续使用 Playwright 而非切换到轻量 HTTP 客户端（如 Cheerio + fetch）？**
- 已在项目中集成，无需引入新依赖
- 支持 JavaScript 动态渲染页面（SPA、React/Vue 文档站）
- 提供完整的浏览器环境，内容提取更准确
- 虽然资源消耗大于纯 HTTP 方案，但本地工具场景可接受

**K2: 为什么爬取与 Documents 功能集成而非独立管理？**
- 复用现有的分块、向量化、搜索基础设施
- 用户在统一的知识库中搜索所有内容（笔记 + 网页），体验一致
- 减少重复开发，降低维护成本
- 通过 `source` 元数据区分来源即可

**K3: 为什么批量爬取采用串行策略，且不对用户暴露并发配置？**
- 本地工具场景下目标站点服务器承受能力未知，高并发易触发反爬
- 串行爬取更稳定，单页失败不影响后续，错误处理简单
- 请求间隔 >= 1 秒，避免触发反爬策略
- 同时活跃浏览器实例硬限 <= 2（每实例约 200MB），防止内存耗尽卡死系统
- 不暴露并发配置给用户，避免误操作导致系统不可用（详见 4.1 批量爬取性能约束）

**K4: 为什么 MVP 不支持定时爬取？**
- 定时爬取需要持久化的任务调度器，增加系统复杂度
- 本地工具场景下，用户不总是保持应用运行
- MVP 阶段聚焦于手动触发的即时爬取体验
- 后续版本可通过系统级 cron 或应用内调度器支持

**K5: 为什么采用 Playwright 有头浏览器模式处理登录，而非实现自动登录？**
- 各网站登录流程差异巨大（用户名密码、OAuth、验证码、双因素认证、SSO 等）
- 自动登录需要针对每个网站编写适配逻辑，维护成本极高
- 有头浏览器模式下用户手动登录，无需处理验证码等复杂交互
- Playwright 的 BrowserContext 支持 Cookie 持久化，一次登录可复用多次
- 认证信息加密存储在本地，安全可控

| 与 Documents 功能集成产生冲突 | 中 | 低 | 通过 source 元数据区分、完善的测试覆盖 |

### 8.3 API 设计草案

**单页面爬取**
```
POST /api/crawl/single
Body: { url, options?: { timeout, waitForSelector, cssSelector }, authProfileId?: string }
Response: { taskId, preview: { title, content, wordCount, url } }
```
> **同步/异步说明**：此 API 为同步阻塞模式，客户端等待响应直至爬取完成（最长等待时间由 `timeout` 控制，默认 30 秒）。
> 超时未完成时返回 `{ taskId, status: "timeout" }`，前端可通过 `GET /api/crawl/tasks/:taskId` 轮询最终结果。
> `authProfileId` 可选，传入后自动携带对应域名的认证信息。若未传入但该域名有已保存的认证状态，系统自动匹配使用。

**站点地图爬取**
```
POST /api/crawl/sitemap/parse
Body: { sitemapUrl }
Response: { urls: string[], total: number }

POST /api/crawl/sitemap/start
Body: { urls: string[], options?: { concurrency, interval, timeout }, authProfileId?: string }
Response: { taskId }

GET /api/crawl/tasks/:taskId
Response: { status, progress: { completed, total, failed }, results: [...] }
```

**递归爬取**
```
POST /api/crawl/recursive/discover
Body: { startUrl, maxDepth, urlFilter? }
Response: { urls: string[], total: number }

POST /api/crawl/recursive/start
Body: { urls: string[], options?, authProfileId?: string }
Response: { taskId }
```

**任务管理**
```
GET /api/crawl/tasks
Response: { tasks: [...] }

POST /api/crawl/tasks/:taskId/retry
Response: { success: true }

DELETE /api/crawl/tasks/:taskId
Response: { success: true }
```

**导入确认**
```
POST /api/crawl/import
Body: { taskId, items: [{ url, title, content, tags? }] }
Response: { imported: number, documentIds: [...] }
```

**认证管理**
```
POST /api/crawl/auth/launch-browser
Body: { url: string, name?: string }  // 登录页面 URL，name 为认证状态的别名
Response: { sessionId: string, message: "请在弹出的浏览器中完成登录" }
Note: 服务端启动 Playwright 有头浏览器，同时建立 SSE 通道推送状态

GET /api/crawl/auth/launch-browser/:sessionId/events (SSE)
Events: { event: "browser_launched" | "login_detected" | "timeout" }
Note: 前端通过 SSE 监听浏览器状态变化，login_detected 表示后端检测到 URL 离开登录页

POST /api/crawl/auth/complete-login
Body: { sessionId: string, name?: string }  // name 为认证状态别名，如"公司 GitLab"
Response: { success: true, authProfile: { id, domain, name, cookieCount } }

POST /api/crawl/auth/cookie
Body: { domain: string, cookie: string, name?: string }
Response: { success: true, authProfile: { id, domain, name } }

POST /api/crawl/auth/header
Body: { domain: string, headerName: string, headerValue: string, name?: string }
Response: { success: true, authProfile: { id, domain, name } }

GET /api/crawl/auth/profiles
Response: { profiles: [{ id, domain, name, type, createdAt, lastUsedAt }] }

DELETE /api/crawl/auth/profiles/:profileId
Response: { success: true }
```
